- **Consistency across tasks**: Open models spike on individual benchmarks but are more variable across the full evaluation suite. Open source models are production-viable for specific, well-defined tasks: code generation, summarization, structured extraction, chat interfaces. They are not yet drop-in replacements for frontier models in agentic workflows, complex multi-tool pipelines, or safety-critical applications.

The gap is closing faster than anyone predicted. A year ago, the best open model was 15-20 points behind on SWE-Bench. Now it is 4 points. At this rate, the gap on coding benchmarks closes by mid-2026.

## Running Models Locally: Quantization, Sampling, and Configuration

Open source models unlock something frontier APIs never will: running entirely on your hardware. No API calls, no rate limits, no data leaving your machine. Here is how to set it up properly.

### Quantization: Why Unsloth Dynamic 2.0 Is the New Default

Not all quantized models are created equal. Standard GGUF quantization treats every layer the same — smash the whole model down to 4-bit or 8-bit uniformly. [Unsloth Dynamic 2.0](https://unsloth.ai) does something smarter: it analyzes every layer individually and assigns per-layer quantization types based on how sensitive each layer is to precision loss.

The methodology uses calibration on 1.5M+ tokens of curated conversational data (not Wikipedia dumps) and optimizes for KL Divergence — the mathematical distance between the quantized model's output distribution and the original full-precision model. The result: smaller files with less quality loss than both standard imatrix quants and QAT (Quantization-Aware Training) quants.

**Grab Unsloth Dynamic 2.0 GGUFs from HuggingFace.** If you see a model available in both standard GGUF and Unsloth Dynamic 2.0 GGUF, always pick Unsloth.

### Sampling Parameters: The 2026 Consensus

Forget top_k and top_p. The sampling consensus in 2026 is **temperature + min_p**.

min_p scales dynamically with model confidence. Instead of cutting off at a fixed percentile (top_p) or fixed count (top_k), min_p sets a minimum probability threshold relative to the most likely token. High-confidence tokens pass easily; low-confidence tokens get filtered proportionally. This solves top_k's fixed cutoff problem and top_p's flat-distribution problem in one parameter.

| Use Case | Temperature | min_p | Repetition Penalty | When to Use |
|----------|------------|-------|-------------------|-------------|
| Code generation | 0.1 - 0.4 | 0.05 - 0.1 | 1.0 - 1.05 | Deterministic output, minimal creativity |
| General assistant | 0.7 - 1.0 | 0.05 - 0.1 | 1.1 - 1.15 | Balanced responses, some variety |
| Creative writing | 0.8 - 1.2 | 0.02 - 0.05 | 1.1 - 1.2 | Maximum variety, accept occasional weirdness |

**Pro tip:** Start with temperature 0.7 and min_p 0.05 for most tasks. Adjust temperature down for code, up for creative. Do not touch repetition penalty unless you see actual repetition issues — over-penalizing kills output quality.

### Ollama Configuration: The Settings That Actually Matter

[Ollama](https://ollama.com) wraps llama.cpp with a simple API and model management layer. These environment variables unlock performance most users leave on the table:

```bash
# Enable flash attention — reduces memory usage for long contexts
export OLLAMA_FLASH_ATTENTION=1

# Quantize the KV cache to q8_0 — saves VRAM at negligible quality cost
export OLLAMA_KV_CACHE_TYPE=q8_0

# Single request at a time — maximizes throughput for single-user setups
export OLLAMA_NUM_PARALLEL=1

# Keep one model loaded — avoids memory fragmentation
export OLLAMA_MAX_LOADED_MODELS=1
```

**Context length**: Ollama defaults to 2048 tokens. For anything useful, set it explicitly in your Modelfile or via the API:

```
/set parameter num_ctx 8192
```

For coding tasks, 8192-16384 is the sweet spot. Going above 32K with quantized models degrades quality noticeably due to accumulated precision loss in the KV cache.

**llama.cpp equivalent flags** (for direct llama-server usage):

```bash
llama-server \
  -m model.gguf \
  --flash-attn \
  --cache-type-k q8_0 \
  --cache-type-v q8_0 \
  -c 8192 \
  -ngl 99 \
  --host 0.0.0.0 \
  --port 8080
```

The `-ngl 99` flag offloads all layers to GPU. On Apple Silicon, this uses Metal acceleration automatically.

### Making Models Think More: Reasoning Models by VRAM

Standard models generate tokens left-to-right without "thinking." Reasoning models like DeepSeek-R1 and QwQ generate an internal chain-of-thought before producing the final answer — trading speed for accuracy on complex problems.

| VRAM | Best Reasoning Model | Notes |
|------|---------------------|-------|
| 8 GB | DeepSeek-R1-Distill-Qwen-7B @ Q4_K_M | Basic reasoning, limited context |
| 16 GB | QwQ-32B @ Q4_K_M or R1-Distill-14B @ Q5_K_M | QwQ is a tight fit but works |
| 24 GB | QwQ-32B @ Q5_K_M or R1-Distill-32B @ Q4_K_M | Comfortable for extended reasoning |
| 48 GB | DeepSeek-R1-Distill-70B @ Q4_K_M | Full-sized reasoning at home |

**When to use reasoning models vs standard models:**

- **Use reasoning models** for: multi-step math, debugging complex code, logic puzzles, planning tasks, anything where "think step by step" actually helps
- **Use standard models** for: code generation, summarization, translation, chat, structured extraction — tasks where speed matters more than depth

**CoT prompting with standard models**: You do not always need a reasoning model. Adding "Think through this step by step before answering" to your prompt with a standard model like Qwen 3.5 or DeepSeek V3.2 can capture 60-70% of the reasoning benefit at standard inference speed.

## Understanding Benchmarks: Why Most Numbers You Read Are Wrong

The single most important thing to know about AI benchmarks in 2026 is that the most-cited one was garbage. SWE-Bench Verified — the benchmark every company used to prove their model was "best at coding" — was retired in February 2026 after an audit revealed 59.4% of its tasks were flawed or unsolvable under fair evaluation conditions.

### How SWE-Bench Verified Died

[OpenAI's February 23, 2026 announcement](https://openai.com/index/why-we-no-longer-evaluate-swe-bench-verified/) found that frontier models could reproduce verbatim gold patches from memory. Their audit of 27.6% of the dataset revealed at least 59.4% of audited problems had flawed test cases that reject functionally correct solutions. The benchmark's tasks had leaked into training data so thoroughly that models were pattern-matching memorized solutions, not actually solving software engineering problems. When you saw "80% on SWE-Bench Verified," you were seeing "80% recall on a contaminated test set."

SWE-Bench Pro is the replacement from the original team, with a current top score of ~23% — which tells you how inflated the old benchmark was.

### SWE-ReBench: What Actually Measures Coding Ability

[SWE-ReBench](https://swe-rebench.com) from TractoAI is the decontaminated alternative:

- **Continuously evolving**: Uses fresh GitHub issues, making contamination structurally impossible
- **5 runs per model**: Reports variance, not just peak performance
- **Standardized scaffolding**: Every model gets the same agent framework, isolating model capability from tooling advantages
- **Centralized evaluation**: No self-reporting — all evaluations run by TractoAI

Current SWE-ReBench top scores: Claude Code 52.9% pass@1, Claude Opus 4.6 and GPT-5.2 tied at 51.7%. These numbers are 25-50 percentage points lower than the retired SWE-Bench Verified scores because they measure actual coding ability on unseen problems.

### Multi-SWE-Bench: The Python Overweighting Problem

[Multi-SWE-Bench](https://arxiv.org/abs/2504.02605) from ByteDance extends software engineering evaluation to 7 languages (Java, TypeScript, JavaScript, Go, Rust, C, C++) with 1,632 human-validated instances.

The finding that should concern everyone: models show limited generalization beyond Python. Claude 3.7 Sonnet scored 43% on multi-language tasks vs 63% on Python-only — a 20-point drop just from switching languages. When someone tells you a model is "great at coding," ask which language. The answer is almost always Python.

### How to Read Benchmark Claims in 2026

A framework for evaluating any model announcement:

1. **Who ran the evaluation?** Self-reported scores are marketing. Third-party scores (Arena, Artificial Analysis, SWE-ReBench) are data.
2. **What version was tested?** Companies benchmark optimized versions and release different ones (the Llama 4 Maverick problem).
3. **Is the benchmark contaminated?** If a model scores suspiciously well on an old benchmark, the benchmark is probably in its training data.
4. **What is the variance?** A single run tells you nothing. SWE-ReBench reports 5 runs for a reason.
5. **Does it generalize?** A model that scores 80% on Python SWE-Bench but 40% on TypeScript is not "80% good at coding."

### When New Models Drop: A 3-Step Evaluation Framework

Open source models release weekly. Here is how to evaluate them without falling for hype:

1. **Check independently verified scores first.** Look at [LMArena](https://lmarena.ai) Arena ELO, [Artificial Analysis](https://artificialanalysis.ai) AAII, and [SWE-ReBench](https://swe-rebench.com). Ignore self-reported benchmarks until independently confirmed.
2. **Check the license and actual weights.** "Open source" means the weights are downloadable and the license permits commercial use. If either condition fails, it is not truly open.
3. **Run it yourself on your specific task.** Benchmarks measure average performance across curated problems. Your use case might be in the model's sweet spot or its blind spot. A 30-minute test on your actual workload is worth more than every benchmark combined.
